Qu’est-ce qu’un logiciel

Un logiciel, ce n’est pas seulement du code, c’est aussi:

  • Des besoins
  • Une gestion de projet
  • Une spécification (fonctionalités, contraintes, facteur de qualité, interface)
  • Une conception: fonctionnelle, architecturale, algorithmes, conception détaillée...
  • Un code source
  • Un exécutable
  • DES TESTS 🧪

Pourquoi tester

  • Protection contre la régression
  • Documentation exécutable
  • Code moins couplé

Types de test

  • Fonctionnels:
    • Est-ce que le logiciel est conforme à la spécification ?
    • Liés à la spécification, à la qualité, à la performance…
  • Non-Fonctionnels:
    • Est-ce que son usage est conforme ?
    • Liés à la configuration, la compatibilité, à la documentation…
  • Structurels:
    • Est-ce que le codage est correct ?
    • Fautes d’implémentation, ou fonctions non prévues.

Niveaux de test

  • Les tests unitaires permettent de tester un composant, ou un bout de code, isolé de ses dépendances.
  • Les tests d’intégration permettent de vérifier que tous les bouts de code isolés fonctionnent bien ensemble.
  • Les tests end2end valident l’application tout entière.
  • Les tests d’acceptation permettent de s’assurer que l’application répond bien au besoin fonctionnel.

Pyramide de tests

Un bon test unitaire

  • Rapide. Il n’est pas rare pour des projets matures de comporter des milliers de tests unitaires. Les tests unitaires doivent durer très peu de temps.
  • Isolé. Les tests unitaires sont autonomes et peuvent être exécutés de manière isolée. Ils ne dépendent d’aucun facteur externe, par exemple un système de fichiers ou une base de données.
  • Renouvelable. L’exécution d’un test unitaire doit être cohérente avec ses résultats. En d’autres termes, il retourne toujours le même résultat si vous ne changez rien entre les exécutions.
  • Autovérification. Le test doit pouvoir détecter automatiquement son état de réussite ou d’échec sans aucune interaction humaine.
  • Délai approprié. Écrire un test unitaire ne doit pas prendre un temps disproportionné par rapport au code testé. Si vous trouvez que le test du code prend beaucoup de temps par rapport à son écriture, optez pour une conception plus facile à tester.

Assertions

  • Assert + CollectionAssert
  • AreEqual(object expected, object actual)
    • Tests whether the specified objects are equal and throws an exception if they are not equal. AreNotEqual
    • Tests whether the specified condition is false and throws an exception if the condition is true. IsTrue
  • IsNull(Object)
    • Tests whether the specified object is null and throws an exception if it is not. IsNotNull
  • ThrowsException<T>(Action)
    • Tests whether the code specified by delegate action throws exact given exception of type T
  • Contains(ICollection, Object)
    • Tests whether the specified collection contains the specified element and throws an exception if the element is not in the collection. DoesNotContain(ICollection, Object)

Assertions

  • ThrowsException<T>(Action, String)
    • Tests whether the code specified by delegate action throws exact given exception of type T with the exact given message String
public Car GetCar(int plateNumber)
{
    if (plateNumber < 1) 
        throw new ArgumentException("Plate must be > 0")
}

[TestMethod]
public void GetCarTest()
{
    //Arrange
    var service = new CarService();
    // Act+Assert
    Assert.ThrowsException<ArgumentException>(() 
        => service.GetCar(-1), "Plate must be > 0");
}
  • Contains(ICollection, Object)
    • Tests whether the specified collection contains the specified element and throws an exception if the element is not in the collection. DoesNotContain(ICollection, Object)

AAA

  • « Arrange » : initialisation du contexte d’exécution du test unitaire (variables, données, dépendances, bouchons…),
  • « Act » : exécution du traitement nécessaire pour la vérification du comportement du code testé,
  • « Assert » : vérification du critère de réussite du test au travers d’une assertion.

public static class MathOperation  
{

    {             
        return (number1 + number2);  
    }

MathOperationTests.cs

[TestMethod]  
public void Task_Add_TwoNumber()  
{  
    // Arrange  
    var num1 = 2.9;  
    var num2 = 3.1;  
    var expectedValue = 6;  

    // Act  
    var sum = MathOperation.Add(num1, num2);  

    //Assert  
    Assert.Equal(expectedValue, sum);  
}

TDD

  • Test Driven Development

  • Red : Vous écrivez un test simple qui échouera avant même d’avoir du code qui l’accompagne. Le test échoue, évidemment, sans le code. Du coup, le test est rouge.

  • Green : Vous écrivez le code le plus simple possible pour faire passer le test, même si le code est un peu ridicule ou simplifié. Le test réussit, il est donc vert.

  • Refactor : le code que vous avez écrit pour faire passer le test est peut être illogique ou trop simple. Il faut faire un refactor de ce code maintenant si possible.

TDD

  1. Ecriture du test (en fonction du comportement attendu du futur code source)
  2. Lancement et échec du test
  3. Ecriture du code source le plus simple possible qui permettra d’obtenir le comportement attendu par le test
  4. Lancement et réussite du test
  5. Refactorisation du code pour l’optimiser
  6. Lancement du test et vérification qu’il réussit toujours malgré la refactorisation

public static int Add(string numbers)
{
    var parts = numbers.Split(',');
    var result = 0;

    foreach (var part in parts)
    {
        int.TryParse(part, out int number);

        if (number <= 1000)
            result += number;
    }

    return result;
}

Exercice 5 : TDD

  • git commit -am “finished ex4”
  • Dans Character créer une méthode GetScore() qui retourne le score général du personnage qui est une somme de :
    • HitPoints + Strength + Defense + Intelligence
    • Si Class = Knight : on double Defense
    • Si Class = Mage : on double HitPoints
    • Si Class = Cleric : on double Intelligence
    • Mais qui limite le score max à 20, c’est à dire que la somme HitPoints + Strength + Defense + Intelligence <= 22
public void GetScoreTest() 
{
    //Arrange
    var character = new Character { HitPoints = 5, Strength = 2, Defense = 1, 
        Intelligence = 4, Class = RpgClass.Mage };

    //Act
    var result = character.GetScore();

    //Assert
    Assert.AreEqual(17, result);
}
